Slide 3: SOLUTION OVERVIEW
--------------------------
5 specialized AI agents working together in a 7-stage workflow:
1. OBSERVE: Health checks, metrics, alerts, logs, dependencies
2. REASON: LLM analyzes incident, decides next action
3. PLAN: Search KB + similar incidents + create fix plan
4. DECIDE: Review plan, route to executor or escalate
5. EXECUTE: 15 fix actions with approval gates and rollback
6. VALIDATE: 6 check types (health, metrics, synthetic, alerts, compliance, baselines)
7. LEARN: Post-incident learning (future enhancement)

Result: Automates ~70% of common incidents from 4+ hours to under 30 minutes

Slide 4: ARCHITECTURE
---------------------
Streamlit Dashboard (8501) → FastAPI (8000) → LangGraph Agents → Core Services → Infrastructure

Four-layer design:
- API Layer: 6 route modules, 23 endpoints
- Agent Layer: 5 agents with typed LangGraph state machines
- Core Services: 14 modules (config, database, vector search, knowledge, action dispatch, approval, monitoring, validation)
- Infrastructure: Docker Compose (7 services), PostgreSQL, Redis, ChromaDB, Prometheus, Grafana

Dual vector DB: ChromaDB for dev, pgvector for production — switch with one config change
Hybrid search fallback: vector + keyword + brute force when vector DB unavailable

Slide 5: AGENTIC WORKFLOW
--------------------------
7-stage state machine with parallel execution paths:

Path A (Standard): OBSERVE → REASON → PLAN → DECIDE → EXECUTE → VALIDATE → CLOSE
Path B (High Severity): Parallel OBSERVE during PLAN and EXECUTE
Path C (Low Severity): Skip PLAN for known patterns
Path D (Escalation): Any stage → human operator
Path E (Rollback): EXECUTE failure → ROLLBACK → RE-PLAN → RE-EXECUTE

All agents use ReAct pattern: Think → Act → Observe → Repeat
Max 10 iterations, 300s timeout, 3 retries per agent

Slide 8: TECHNICAL DEEP DIVE
----------------------------
Stack: Python 3.12+, LangGraph 1.2.9, LangChain 0.3.86, FastAPI, Pydantic v2

AI Integration:
- Default: NVIDIA Nemotron via OpenRouter
- Configurable: OpenAI, Anthropic, Ollama (local) — one config change
- Rule-based fallback when LLM unavailable (graceful degradation)

Vector Search:
- ChromaDB (development) and pgvector (production)
- Automatic fallback to seed corpus + brute force comparison
- Embedding dimension: 4096 (text-embedding-3-small default)

Safety:
- Dry-run mode via feature flag
- Human approval gates for 8 risky action types
- Automatic rollback procedures for every remediation step
- Confidence scoring for all AI decisions
- 12 feature flags for gradual rollout
- Structured logging with structlog (auto-fallback to stdlib)

Testing: 93 unit tests across 8 test files

Slide 11: DIFFERENTIATION
--------------------------
What makes Aegis different from existing tools:

PagerDuty/ServiceNow/Opsgenie — Route incidents to people. Aegis FIXES them automatically.

PagerDuty AIOps — Single AI assistant. Aegis uses 5 specialized agents working as a team.

Rule-based automation — Fixed if-then-else. Aegis uses LLM reasoning for novel situations.

Chatbots — Answer questions only. Aegis takes action: restart, scale, rollback, block, isolate.

Open source (MIT) — No vendor lock-in, deploy in 5 minutes. Free vs $100K+/year proprietary.

Built-in validation — 6 check types verify fixes worked. Most tools stop at execution.

Human-in-the-loop — Intelligent approval gates, not all-or-nothing automation.

Dual vector DB + hybrid search — Works with or without vector database infrastructure.
